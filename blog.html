Building AI Agents That Actually Work in Production

June 2026

AI agent demos are everywhere. A chatbot books a flight, a coding assistant writes a feature, an automation agent processes invoices — all in a polished two-minute video. But anyone who has tried to deploy these systems into production knows the gap between demo and reality is enormous.

At Arvi AI Tech, we have built and deployed multi-step AI agents for customer support, internal workflow automation, and game content generation. Here is what we have learned about making them reliable enough for production use.

The Demo-to-Production Gap

In a demo, the agent handles the happy path perfectly. In production, it encounters ambiguous instructions, missing data, API timeouts, rate limits, and edge cases no one anticipated. The difference between a demo agent and a production agent is the same as the difference between a script and a system. An agent's execution path depends on LLM outputs, which are probabilistic. Every step is a potential failure point, and failures cascade in unpredictable ways.

Lesson 1: Design for failure at every step

Every tool call can fail, every LLM response can be malformed, every external API can timeout. We wrap each tool call in structured error handling and feed clear error messages back to the LLM — which is often your best error handler, finding workarounds you never coded explicitly.

Lesson 2: State management is everything

The most common production failure is lost or corrupted state. We checkpoint state after every successful step to a durable store, so an agent can resume from where it failed rather than starting over.

Lesson 3: Observability is not optional

Every agent emits structured logs at every decision point — what was asked, what it decided, what tools it called, and how long each step took. This catches subtle bugs where agents technically succeed but take expensive or slow paths.

Lesson 4: Constrain the action space

The most reliable agents have carefully limited capabilities. We follow least privilege — read-only access where possible, confirmation steps for irreversible actions, rate limits on expensive operations.

Lesson 5: Test with chaos, not just happy paths

We maintain adversarial test scenarios — APIs that fail on the third call, responses that are valid but semantically wrong, contradictory user inputs. These catch regressions that happy-path tests miss.

Building production AI agents is a systems engineering challenge, not an AI challenge. The LLM is the easy part. The hard part is state management, error handling, observability, and testing. Treat agents as distributed systems, and they become remarkably reliable.

Procedural Level Generation: Making Every Playthrough Unique

June 2026

One of the hardest problems in puzzle game design is content. Players consume levels faster than designers can create them. Procedural generation promises infinite content, but naive approaches produce levels that are unsolvable, trivially easy, or frustratingly unfair.

The solvability guarantee

Every generated level must be solvable — a player hitting an impossible level loses trust permanently. Our approach is construction-based, not generation-and-test. We build levels by working backwards from a solved state, applying reverse moves to create the puzzle. This guarantees solvability by construction, because we can always replay the shuffle in reverse.

Difficulty control

Solvability isn't enough; levels need the right difficulty. We control it through parameters: minimum solution length, number of decision points, presence of dead ends, and the ratio of constrained to free moves. The generator produces candidates and keeps only those matching the target difficulty profile — fast, because it already knows the solution.

The interestingness problem

A solvable level at the right difficulty can still be boring. Interesting puzzles have structure — recognizable patterns, moments of insight, satisfying state evolution. We bias generation toward configurations that create clusters, bottlenecks, and cascades. These biases are tunable per game and difficulty tier.

Performance at scale

Our generators run in the build pipeline for level packs and on-device for endless mode. On-device generation targets under 50 milliseconds on mid-range mobile hardware, using lookup tables and precomputed move sequences rather than expensive graph searches.

Procedural generation is not about randomness — it's about controlled variation within carefully designed constraints. The constraints encode game design knowledge; the variation ensures no two levels feel the same.

A Practical Guide to Building RAG Systems

May 2026

Retrieval-Augmented Generation lets you build AI features grounded in your own data — up-to-date and verifiable. But building a RAG system that works in production is harder than the tutorials suggest.

Chunking is the foundation

The most impactful decision is how you chunk documents. Too large wastes context; too small loses meaning. For technical docs we use 400–600 token chunks with 50–100 token overlap; for support tickets, 200–300 tokens; for legal documents, 800–1000. How you split matters as much as size — we use semantic chunking that respects paragraph boundaries and section headers, never splitting mid-sentence.

Embedding model selection

The embedding model determines retrieval quality. We evaluate models on our clients' actual query-document pairs, not public benchmarks, using sets of 100–200 rated pairs before committing.

Retrieval is not just vector search

Production RAG needs hybrid retrieval combining vector and keyword search, metadata filtering to narrow the search space, and a reranking layer. After initial retrieval returns 20–50 candidates, a cross-encoder reranker scores each against the query — the top results after reranking are far more relevant than vector search alone.

The generation step

Instruct the model to cite sources (makes responses verifiable and helps debug retrieval). Tell it to say "I don't have enough information" when chunks lack the answer, or it will hallucinate. Place the most relevant chunks first — LLMs pay more attention to the start and end of context.

Evaluation and monitoring

RAG systems degrade silently. We track retrieval relevance, answer faithfulness, and answer completeness using a mix of automated LLM-as-judge and human review.

Start simple. Good chunking, a standard embedding model, and a well-engineered prompt beat a complex system with poor fundamentals. Add complexity only when metrics show where the simple system falls short.

Adaptive Difficulty: How AI Keeps Players in the Flow

May 2026

The best games feel like they read your mind. When you struggle, the challenge eases; when you cruise, it ramps up. This is adaptive difficulty, and modern AI makes it far more effective than the simple approaches of the past.

What is flow

Flow is full immersion at the edge of your ability. Too easy, you get bored; too hard, you get frustrated. Traditional Easy/Normal/Hard settings are a blunt instrument — they assume all players in a tier have the same skill, and that skill never changes. Both are wrong.

Player modeling

We model players across multiple dimensions, not a single skill number. A player might excel at spatial reasoning but struggle under time pressure. Our models track completion rate, time, hint usage, retries, and move efficiency per mechanic, updating after every level with exponential smoothing to capture improvement.

Difficulty adjustment

Adjustment should be invisible — players should feel they're getting better, not that the game got easier. We tune structural elements players don't directly perceive (red herring moves, solution length, proximity to the goal) rather than obvious knobs like timers. And we adjust in both directions, increasing difficulty as skill grows.

The cold start problem

New players are hardest to model. We solve it with a calibration sequence disguised as a tutorial — each early level secretly tests a specific skill dimension while teaching mechanics. The model converges within 15–20 levels.

Measuring success

The metrics that matter are session length, voluntary return rate, and quit-point distribution — plus the share of players rating difficulty as "just right." When adaptive difficulty works, it's invisible; players just feel the game is well-designed.

Why We Use Flutter for Casual Game Development

April 2026

When people think of game development they think of Unity or Unreal — powerful engines built for 3D worlds. But for casual and puzzle games that dominate mobile app stores, they're often overkill. At Arvi AI Tech, Flutter has become our primary framework for 2D casual games.

The case for Flutter

One codebase runs on iOS, Android, and web with native performance. The Flame engine provides sprite rendering, animation, collision detection, and particle effects, integrating seamlessly with Flutter's widget system — so game UI uses the same framework as the game. Hot reload and Dart's tooling let us ship in weeks what would take months in Unity.

Performance is good enough

Flutter renders at 60fps on modern devices, and our puzzle games run at 60fps on hardware three to four years old. The caveats are older devices with heavy transparency effects, which we mitigate with early profiling.

Where Flutter falls short

3D is its weakest area — use a dedicated 3D engine for that. Complex physics (ragdolls, fluids, destructible environments) needs a real physics engine. Audio is adequate for sound effects and music but not for spatial or procedural audio.

Our stack

Flame for the engine, Riverpod for state, Hive for persistence, Firebase for analytics, Rive or Lottie for animation, and custom procedural generators for levels. Mature, well-documented, and fast for new team members.

Flutter isn't trying to replace Unity — it occupies a different niche where cross-platform reach and development speed matter more than cutting-edge graphics. For puzzle, tap, card, and word games, it delivers everything we need at a fraction of the cost.

Architecting LLM-Powered Applications: Lessons from the Field

April 2026

LLMs are the most versatile building blocks in modern software, but integrating them into production requires a different mindset. Their probabilistic outputs, cost structure, latency, and failure modes are all different from deterministic APIs.

Prompt management

Prompts are code — version-controlled, tested, and reviewed. We store them as templated strings separate from logic, with test cases per prompt and changelogs that record what changed and why. When quality drops, the changelog points to the culprit.

Output validation

Never trust LLM output. For structured output we validate against a schema and retry with an error message on failure. For unstructured output we use lightweight classifiers to catch hallucinated URLs, fabricated statistics, or off-topic responses.

Cost control

The biggest cost drivers are unnecessary calls and oversized prompts. A semantic cache can eliminate 30–50% of calls. Trim retrieved documents to relevant excerpts, summarize long conversation history, and use model routing — sending simple queries to cheaper models — to cut costs 40–60%.

Latency optimization

Streaming is the most impactful: time-to-first-token under 500ms feels responsive even if the full response takes seconds. Make independent LLM calls in parallel, and use speculative execution for predictable workflows.

Testing strategy

LLM apps can't be tested with exact assertions. We use evaluation-based testing against quality criteria and maintain golden datasets — curated inputs with human-rated reference outputs — as regression tests that block deployment on quality drops.

The architecture we've converged on is a simple pipeline: input validation → prompt construction → model routing → LLM call with retries → output validation → post-processing. Each stage is independently testable and observable. Simplicity is the point — LLM applications have enough inherent complexity already.
